## Задание 3: Сравнение производительности CPU vs GPU (`homework_performance.py`)

### Методика

- Созданы три набора матриц размеров: `64×1024×1024`, `128×512×512`, `256×256×256`.  
- Для каждой размерности замерялось время выполнения пяти операций:  
  - матричное умножение;  
  - поэлементное сложение;  
  - поэлементное умножение;  
  - транспонирование;  
  - вычисление суммы всех элементов.  
- Измерения проводились на CPU (с помощью `time.perf_counter`) и на GPU (с помощью `torch.cuda.Event`).  
- Для каждой операции вычислено ускорение (speedup = время_CPU / время_GPU).

## Анализ результатов

### 1. Наибольшее ускорение

- **Поэлементные операции (сложение, умножение)** показывают стабильно высокое ускорение – от **46× до 62×** (в зависимости от размера). Это объясняется тем, что они идеально параллелизуются: каждый элемент обрабатывается независимо, и GPU может задействовать тысячи ядер одновременно.
- **Матричное умножение** также демонстрирует значительный прирост – **от 30× до 44×**. Хотя оно сложнее поэлементных операций, специализированные библиотеки (cuBLAS) эффективно используют тензорные ядра и иерархию памяти GPU.

### 2. Операции с низким ускорением

- **Транспонирование** показывает ускорение **меньше 1×** (вплоть до 0.31×), то есть на GPU оно выполняется **медленнее**, чем на CPU. Причина в том, что транспонирование – это, по сути, перестановка данных в памяти без арифметических вычислений. Скорость такой операции ограничена пропускной способностью шины памяти, а накладные расходы на запуск ядра CUDA и синхронизацию перевешивают выигрыш от параллелизации.
- **Суммирование** (редукция) даёт ускорение около **20–23×** – меньше, чем у поэлементных операций. Это связано с тем, что редукция требует глобальной синхронизации между блоками, что снижает эффективность параллельной обработки.

### 3. Влияние размера матриц

- С ростом размера данных **ускорение для вычислительно-интенсивных операций (умножение, сложение)** остаётся высоким, но не монотонно растёт. Например, для матричного умножения ускорение колеблется: 38.8× (64×1024×1024), 30.0× (128×512×512), 36.7× (256×256×256). Это может быть связано с эффективностью использования кэша и загрузкой ядер – оптимальный размер зависит от архитектуры GPU.
- Для транспонирования ускорение остаётся <1× для всех размеров, что подтверждает, что данная операция невыгодна для GPU независимо от объёма данных.

### 4. Передача данных между CPU и GPU

В проведённом эксперименте мы измеряли только время выполнения операций над уже размещёнными на устройствах тензорами. Однако в реальных задачах время копирования данных из оперативной памяти в видеопамять (и обратно) может быть значительным. Например, для размера `64×1024×1024` (около 67 млн элементов типа float32, ~268 МБ) передача может занять десятки миллисекунд, что сопоставимо со временем вычислений. Поэтому на практике рекомендуется:
- как можно дольше держать данные на GPU;
- выполнять все преобразования на GPU без лишних перемещений;
- использовать асинхронные копирования и перекрытие вычислений с передачей данных (если это поддерживается).

### 5. Сводная таблица по типам операций

| Тип операции       | Характер ускорения          | Причина                                   |
|--------------------|-----------------------------|-------------------------------------------|
| Матричное умножение| Высокое (30–44×)            | Интенсивные вычисления, хорошая параллелизация |
| Поэлементное       | Очень высокое (46–62×)      | Полная независимость элементов, максимальный параллелизм |
| Транспонирование   | Низкое (<1×)                | Работа с памятью, накладные расходы на запуск ядер |
| Суммирование       | Умеренное (20–23×)          | Требует синхронизации, но всё же выигрывает за счёт параллельной редукции |

### Общий вывод

GPU даёт огромный выигрыш в скорости для задач, насыщенных арифметикой (умножение матриц, поэлементные преобразования), но практически бесполезен (или даже вреден) для простых операций перестановки данных, таких как транспонирование. Размер данных влияет на ускорение, но не является главным фактором – важнее характер операции. Для достижения максимальной производительности необходимо минимизировать обмен данными между CPU и GPU и выбирать подходящие алгоритмы, хорошо использующие параллельную архитектуру.
